# Task Format Standard

> Gold standard: `clients/example-ecom/TASKS.md`
> Every task across every project MUST follow this format.
> Added: 2026-03-26

## Required Format

```markdown
- [ ] #{N} {Title} — {brief description if title isn't self-explanatory}
  - @{gtd_tag} @{context_tag}
  - {Priority} | Owner: {Name} | Due: {YYYY-MM-DD} | Effort: {estimate} | Impact: {scale}
  - Created: {YYYY-MM-DD} | Updated: {YYYY-MM-DD}
  - Layer: {L0-L7}
  - {Additional context lines — description, notes, blockers}
  - ext: {external_system_id} | synced: {YYYY-MM-DD}
```

## Field Reference

### Line 1: Status + Number + Title
```
- [ ] #114 Egykapus rendszer kialakítása — [Name]ral dabasi irodában
```
- `- [ ]` = open, `- [x]` = done
- `#{N}` = sequential number within this project's TASKS.md
- Title in the project's working language (HU for HU clients, EN for EN)

### Line 2: GTD Tags
```
  - @next @computer
```
- **Required:** at least one GTD tag
- Tags: `@next` `@waiting` `@someday` `@reminder` `@monitor` `@decision` `@reference` `@milestone`
- Context tags: `@computer` `@phone` `@meeting` `@deep` (optional)

### Line 3: Metadata (ALL REQUIRED)
```
  - P0 | Owner: [Owner] + [Name] | Due: 2026-03-26 | Effort: 4h | Impact: unlock
```
- **Priority:** P0 (critical), P1 (high), P2 (medium), P3 (low)
- **Owner:** Name(s) who DOES the work. Use short names from TEAM_DIRECTORY.md
- **Due:** YYYY-MM-DD or `W{N}` for week-level. Omit only for @someday tasks.
- **Effort:** `5m`, `15m`, `30m`, `1h`, `2h`, `4h`, `1d`, `2d`, `1w`, `2w+`
- **Impact:** `noise` → `hygiene` → `lever` → `unlock` → `breakthrough`

### Line 3b: Origin & Stakeholders (RECOMMENDED)
```
  - From: [Client Contact] (meeting 2026-03-23) | Inform: [Name], [Stakeholder]
```
- **From:** WHO originated/requested this task. Person name + source (meeting, email, Slack, task extraction).
- **Inform:** WHO needs to know when this is done. People who didn't request it but care about the outcome.

**Why this matters:**
- When a task is done → you know WHO to tell (Inform list)
- When a task is blocked → you know WHO to escalate to (From = the person who wanted it)
- When prioritizing → tasks from [Client Contact] (CEO) hit different than tasks you created yourself
- `/task-sync` can add Inform people as [Task Manager] followers / Asana followers

**From vs Owner vs Inform:**
| Field | Who | Example |
|-------|-----|---------|
| **From** | Requested/originated the task | [Client Contact] (asked for it in meeting) |
| **Owner** | Does the work | [Owner] + [Name] |
| **Inform** | Needs to know when done | [Stakeholder], [Team Member 1] |

**Examples:**
```markdown
- [ ] #115 Hirdetési keret növelés — napi +$500
  - @waiting
  - P0 | Owner: [Owner] + [Name] | Due: 2026-03-27 | Effort: 1h | Impact: unlock
  - From: [Client Contact] (meeting 2026-03-23) | Inform: [Stakeholder], [Agency A]
  - Created: 2026-03-24 | Updated: 2026-03-26
  - Layer: L5
  - Domain: example-d2c.com

- [ ] #175 example-pool.com form csere → Jotform + AC integráció
  - @next @computer
  - P1 | Owner: [Owner] | Due: 2026-04-08 | Effort: 4h | Impact: unlock
  - From: [Owner] (analysis/JOTFORM-EVALUATION.md) | Inform: Adrián, [Name]
  - Created: 2026-03-25 | Updated: 2026-03-25
  - Layer: L6
  - Domain: example-pool.com

- [ ] #62 ExampleRetail business analysis — large basket decline
  - @next @computer
  - P0 | Owner: [Owner] + [Team Member 1] | Due: 2026-04-01 | Effort: 4h | Impact: lever
  - From: Pali (observed basket decline) | Inform: Andris, Dóri
  - Created: 2026-03-25 | Updated: 2026-03-25
  - Layer: L7
  - Domain: example-retail.com
```

**Common From sources:**
- `From: {name} (meeting {date})` — came from a meeting
- `From: {name} (email {date})` — came from an email
- `From: {name} (Slack)` — came from Slack
- `From: audit FND-{N}` — generated from audit finding
- `From: /council diagnostic {date}` — generated by council deliberation
- `From: self` — you created it yourself (still useful to track)

### Line 4: Timestamps (REQUIRED)
```
  - Created: 2026-03-24 | Updated: 2026-03-24
```
- When done: add `| Completed: {date}` or `| Done: {date}`

### Line 5: Layer (MANDATORY)
```
  - Layer: L5
```
- **EVERY task MUST have a Layer.** No exceptions.
- Multiple layers OK: `Layer: L5, L7`
- L0=Source/Beliefs, L1=Core/Capacity, L2=Identity, L3=Product, L4=Offer, L5=Channels, L6=Customer, L7=Market

### Line 6+: Dependencies & Waiting (REQUIRED for @waiting tasks)

**For @waiting tasks — these fields are MANDATORY:**

```
  - Waiting on: {Person name} — {what we're waiting for}
  - Waiting since: {YYYY-MM-DD}
  - Follow-up: {YYYY-MM-DD} — {action if no response}
```

Example:
```
- [ ] #115 Hirdetési keret növelés — napi +$500
  - @waiting
  - P0 | Owner: [Owner] + [Name] | Due: 2026-03-27 | Effort: 1h | Impact: unlock
  - Created: 2026-03-24 | Updated: 2026-03-26
  - Layer: L5
  - Waiting on: [Stakeholder] + [Agency A] — keret jóváhagyás és beállítás
  - Waiting since: 2026-03-26
  - Follow-up: 2026-03-28 — [Owner] emlékeztető [Name]nak ha nincs válasz
  - Depends on: #113
```

**Fields:**
- **Waiting on:** WHO we're waiting for + WHAT we're waiting for. Always a person name, never a role.
- **Waiting since:** When we started waiting. The Task Overseer uses this to flag stale waits (>7 days).
- **Follow-up:** When to poke if no response. What to do. Prevents tasks from rotting in @waiting forever.

**For dependency chains:**

```
  - Depends on: #113, #117 — {brief description of what must finish first}
  - Blocks: #20, #116 — {what can't start until this is done}
```

The Task Overseer traces dependency chains: if #113 is overdue and #115 depends on it, both get flagged together.

### Line 7+: Domain Scope (RECOMMENDED for multi-domain clients)

Most clients have multiple domains (ExampleBrand: 8+, ExampleRetail: 2, DH: 5+). Tasks often apply to specific domains, not the whole client. Use the `Domain:` field to scope tasks:

```
  - Domain: example-d2c.com
```

Multiple domains:
```
  - Domain: example-d2c.com, examplebrandfactorystore.com
```

All domains:
```
  - Domain: all
```

**Why this matters:**
- `/task-oversight` can group tasks by domain → "example-d2c.com has 12 open tasks, example-b2b.com has 3"
- When filtering tasks for a specific meeting (e.g., "what's open on exampled2c?"), the domain field enables precise filtering
- GTM container and GA4 property mapping depends on knowing which domain a task targets
- SEO, measurement, and campaign tasks are almost always domain-specific

**Domain values come from CLIENT_CONFIG.md.** Use the exact domain string from the config file. If a task applies to a market (not a specific domain), use the market: `Domain: US` or `Domain: HU`.

Examples:
```markdown
- [ ] #113 ASAP összekötések — sGTM, BigQuery, GTM admin
  - @next @computer
  - P0 | Owner: [Owner] + [Team Member] | Due: 2026-03-26 | Effort: 1d | Impact: unlock
  - Created: 2026-03-24 | Updated: 2026-03-24
  - Layer: L5
  - Domain: example-d2c.com
  - SOP: 08-MEASUREMENT-SETUP

- [ ] #138 Semrush — hozzáférés és beállítás
  - @next @computer
  - P1 | Owner: [Owner] | Due: 2026-03-29 | Effort: 1h | Impact: lever
  - Created: 2026-03-24 | Updated: 2026-03-26
  - Layer: L5
  - Domain: example-d2c.com, example-b2b.com, example-ecom.com, examplebrand.fr, examplebrand.nl, examplebrand.com, examplebrandfactorystore.com
  - SOP: 07-VENDOR-ACCESS

- [ ] #175 example-pool.com form csere → Jotform + AC integráció
  - @next @computer
  - P1 | Owner: [Owner] | Due: 2026-04-08 | Effort: 4h | Impact: unlock
  - Created: 2026-03-25 | Updated: 2026-03-25
  - Layer: L6
  - Domain: example-pool.com
  - SOP: 02-LEAD-LIFECYCLE
```

### Line 8+: Other Context (recommended)

```
  - SOP: 07-VENDOR-ACCESS
  - Meeting: 2026-03-23 Heti [Client Contact] examplebrand
  - FND: FND-009
  - Goal: Q1-examplebrand-measurement
  - Related: #172 (US/ExampleD2C promóciós naptár)
```
- **SOP:** link to relevant SOP (strongly recommended for operational tasks)
- **Meeting:** source meeting if task came from a meeting
- **FND/REC:** finding or recommendation reference
- **Goal:** which quarterly goal this serves
- **Related:** informational link (not a dependency)

### Last Line: External Sync (if synced)
```
  - ext: {[task-manager]-project-id} | synced: 2026-03-24
```
- `ext:` = [Task Manager] task ID or Asana task GID
- `synced:` = last sync timestamp

## Done Tasks

When completed, change `- [ ]` to `- [x]`, add completion date, then move to TASKS_DONE.md:

```markdown
- [x] #114 Egykapus rendszer kialakítása — [Name]ral dabasi irodában
  - @done
  - P0 | Owner: [Owner] + [Name] | Effort: 4h | Impact: unlock
  - Created: 2026-03-24 | Completed: 2026-03-26
  - Layer: L5
  - Meeting: 2026-03-23 Heti [Client Contact] examplebrand
```

## Validation Checklist (Task Overseer uses this)

A task is VALID if ALL of these are true:
- [ ] Has `#{N}` task number
- [ ] Has at least one GTD tag (@next, @waiting, etc.)
- [ ] Has Priority (P0-P3)
- [ ] Has Owner
- [ ] Has Impact scale
- [ ] Has Created date
- [ ] Has Layer (L0-L7)
- [ ] If @next or @waiting: has Due date
- [ ] If @waiting: has `Waiting on:` (person + what)
- [ ] If @waiting: has `Waiting since:` (date)
- [ ] If @waiting: has `Follow-up:` (date + action)
- [ ] If has `Depends on:`: the dependency tasks exist and are not done

A task SHOULD also have:
- [ ] `From:` (who originated) + `Inform:` (who to tell when done)
- [ ] `Domain:` field (for clients with 2+ domains — scope where this task applies)
- [ ] SOP reference (for operational/audit tasks)
- [ ] Meeting reference (if sourced from a meeting)
- [ ] Effort estimate
- [ ] `Blocks:` field if other tasks depend on it
- [ ] ext: + synced: (if project syncs to [Task Manager]/Asana)
