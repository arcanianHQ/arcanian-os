# GTM Container Data Standard

> How we store, version, and track GTM/sGTM container exports across all clients.

---

## Storage Location

```
clients/{slug}/data/gtm-exports/
├── GTM-{CONTAINER_ID}_workspace{N}.json    ← Individual exports (versioned by workspace #)
├── GTM_CHANGELOG.md                         ← What changed between versions
└── GTM_INVENTORY.md                         ← Container summary (tags, triggers, variables count)
```

**NOT in:** `audit/gtm_v1/` or `audit/gtm_samples/` — those are historical/reference. All active exports go in `data/gtm-exports/`.

## Naming Convention

```
GTM-{CONTAINER_ID}_workspace{N}.json
```

Examples:
- `GTM-EXAMPLE-001-005_workspace52.json` — ExampleLocal web, workspace 52
- `GTM-EXAMPLE-001-006_workspace20.json` — ExampleLocal sGTM, workspace 20
- `GTM-EXAMPLE-001-003_workspace53.json` — ExampleAuto web
- `GTM-EXAMPLE-001-004_workspace19.json` — ExampleAuto sGTM

**Rules:**
- Container ID = exact GTM ID (case-sensitive)
- Workspace number = sequential (higher = newer)
- Never rename — the filename IS the version

## Ontology Integration

GTM exports connect to:
- `CLIENT_CONFIG.md` → Container IDs listed
- `audit/findings/FND-NNN` → findings reference container + workspace
- `TASKS.md` → tasks reference findings which reference containers
- `GTM_CHANGELOG.md` → what changed and why (links to FND/REC/Task)

## When to Export

| Event | Action |
|---|---|
| Before any audit (Phase 3) | Export latest workspace → `data/gtm-exports/` |
| After publishing changes | Export new workspace → log in GTM_CHANGELOG |
| Weekly (if active development) | Export for versioning |

## Migration Note

Old exports in `audit/gtm_v1/` and `audit/gtm_v1.1/` are historical.
Don't move them — they're referenced by findings. But new exports go to `data/gtm-exports/`.
