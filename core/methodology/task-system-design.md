## 2. Six Task Source Types Identified

### A. Findings (FND) ‚Äî WELL CONNECTED
- **Where:** `[audit-framework]/clients/*/findings/FND-NNN_*.md`
- **Chain:** FND ‚Üí REC ‚Üí Task (cross-referenced in TASKS.md via `Related: FND-xxx, REC-xxx`)
- **Projects:** ExampleRetail, ExampleLocal
- **Status:** This is the gold standard. Other source types should aim for this level of traceability.

### B. SOPs / Processes ‚Äî NOT CONNECTED
- **Where:** `_example_full/processes/00-21` (21 numbered process docs)
- **Where:** `[audit-framework]/methodology/MEASUREMENT_HEALTH_SOP.md`
- **Where:** `__example_full/skills/` (34 skills = methodology processes)
- **Gap:** Tasks don't reference which SOP/process they follow or were generated by.
- **Need:** `Source: SOP-01` or `Process: 05-CAMPAIGN-MANAGEMENT` in task metadata.
- **Two directions:**
  - Process **generates** recurring tasks (e.g., "every month run 04-MEASUREMENT-REPORTING")
  - Task **references** which process it follows (e.g., "onboard [Agency A] replacement ‚Üí follows SOP 01")

### C. Calendar / Meetings / Events ‚Äî WEAK CONNECTION
- **Where:** Inline in ExampleBrand tasks ("IT vezet≈ë meeting ‚Äî [Name], 03.10 h√©tf≈ë")
- **Where:** `memory/events.md` (AC London, PODIM 2026)
- **Gap:** Date is in the note text, not a structured field. No link to calendar event.
- **Need:** `Meeting: 2026-03-10 IT vezet≈ë` or similar structured reference.

### D. Emails ‚Äî WEAK CONNECTION
- **Where:** Inline in ExampleBrand tasks ("[Former Team Member] email deactivation risk")
- **Gap:** Mentioned as context but not traceable to a specific email/thread.
- **Need:** At minimum, `Email: subject line / date / sender` reference.

### E. Goals (Quarterly / Yearly / Personal / Business) ‚Äî NOT CONNECTED
- **Where:** `_ExampleBuild_full/DELUXE_OKR_BLUF_2026_01_20.md`
- **Where:** `__example_full/arcanian/archive/TASKS_2026_Q1.md` (phase-based goals)
- **Where:** [Task Manager] (goals feature ‚Äî couldn't verify, API down)
- **Gap:** Goal documents exist but tasks don't reference which goal they serve.
- **Need:** `Goal: Q1-ship-[diagnostic-service]` or `OKR: revenue-target` in task metadata.
- **Cascade question:** Goal ‚Üí Key Result ‚Üí Initiative ‚Üí Task? Or just Goal ‚Üí Task?

### F. GTD Tags ‚Äî NOT CONNECTED LOCALLY
- **Where:** [Task Manager] labels (couldn't pull ‚Äî API 401)
- **Gap:** No local representation of GTD contexts (@waiting, @next, @someday, @context).
- **Need:** Tags/labels field in local task format.

## 7. Architectural Insight: Palantir Ontology Layer Analogy

> "In an ideal world, all these things would be connected to a task ‚Äî like the Palantir ontology layer manages."

The Palantir Foundry Ontology is the right mental model. It doesn't store data ‚Äî it **maps relationships between objects**. Applied here:

### Object Types (our ontology)
```
Task ‚Üê‚Üí Finding (FND)
Task ‚Üê‚Üí Recommendation (REC)
Task ‚Üê‚Üí Process / SOP
Task ‚Üê‚Üí Calendar Event / Meeting
Task ‚Üê‚Üí Email Thread
Task ‚Üê‚Üí Goal (quarterly / yearly / personal / business)
Task ‚Üê‚Üí Person (owner, stakeholder)
Task ‚Üê‚Üí Project (client / internal)
Task ‚Üê‚Üí GTD Context (@next, @waiting, @someday)
```

### What this means for the local .md format:
- A task is not just "do X by Y" ‚Äî it's a **node with typed edges** to other objects
- The local file is not a flat list ‚Äî it's a **lightweight ontology serialized as markdown**
- Each reference field (`FND:`, `SOP:`, `Meeting:`, `Goal:`, `Email:`) is an **edge** in the graph
- This is why the ExampleRetail FND‚ÜíREC‚ÜíTask chain works so well ‚Äî it's already ontological

### Palantir parallel:
| Palantir Concept | Our Equivalent |
|---|---|
| Object Type | Task, Finding, Process, Goal, Meeting, Email |
| Property | Priority, Status, Due, Owner |
| Link | `FND: FND-039`, `SOP: 05-CAMPAIGN`, `Goal: Q1-ship-[diagnostic-service]` |
| Action | Status change, completion, escalation |
| Ontology View | The local TASKS.md file = a view over the graph |

## 8. GTD Label System (CONFIRMED)

GTD-style labels are the tagging system. These map to [Task Manager] labels AND local .md tags.

### Core GTD Labels
| Label | Meaning | When to use |
|---|---|---|
| `@next` | Next physical action | Ready to do, no blockers |
| `@waiting` | Waiting for someone/something | Delegated or blocked by external |
| `@someday` | Someday/maybe | Not committed, review monthly |
| `@agenda` | Discuss at next meeting | Needs face-to-face or call |
| `@review` | Needs review/decision | Requires thinking, not doing |

### Context Labels (WHERE/HOW)
| Label | Meaning |
|---|---|
| `@computer` | Needs laptop/desk |
| `@phone` | Can do by phone |
| `@email` | Requires sending/reading email |
| `@errand` | Physical location needed |

### Energy/Time Labels (optional, GTD-adjacent)
| Label | Meaning |
|---|---|
| `@deep` | Needs focus time (>30 min) |
| `@quick` | Can do in <5 min |

### How labels interact with source types:
- A task can be `@waiting` AND linked to `Email: Bacsa NIS2 reply`
- A task can be `@next` AND linked to `SOP: 01-AGENCY-COORDINATION`
- A task can be `@someday` AND linked to `Goal: Q2-[diagnostic-service]-productize`

## 9. Design Requirements (Emerging)

A unified local task format needs to support:

1. **Priority** (P0-P3) ‚Äî proven in ExampleRetail/ExampleLocal
2. **Status** (open, in-progress, done, blocked, wont-do)
3. **Source references (ontology edges):**
   - `FND: FND-039` (finding)
   - `REC: REC-039` (recommendation)
   - `SOP: 05-CAMPAIGN-MANAGEMENT` (process)
   - `Meeting: 2026-03-10 IT vezet≈ë` (calendar event)
   - `Email: [Former Team Member] handover 2026-03-03` (email thread)
   - `Goal: Q1-ship-[diagnostic-service]` (goal/OKR)
4. **GTD labels** ‚Äî `@next`, `@waiting`, `@someday`, `@agenda`, `@review`, `@computer`, `@phone`, `@email`, `@deep`, `@quick`
5. **Owner** ([Owner], [Team Member 1], [Team Member 2], [Name], ITG, client)
6. **Due date**
7. **Project** (ExampleRetail, ExampleBrand, ExampleBuild, Arcanian, ExampleLocal)
8. **[Task Manager] sync ID** (when available)

## 10. Decisions Made

- [x] **GTD labels:** YES ‚Äî use GTD-style labels (confirmed 2026-03-23)
- [x] **No recurring tasks:** Tasks are one-off. SOPs don't auto-generate recurring tasks. (confirmed 2026-03-23)
- [x] **Ontology model:** Palantir-style ‚Äî task is a node with typed edges to other objects (confirmed 2026-03-23)
- [x] **One file per project:** Each project gets its own TASKS.md in its project root. (confirmed 2026-03-23)
- [x] **Checkbox format:** Checkboxes over tables ‚Äî faster to scan, easier to edit. Metadata on indented lines. (confirmed 2026-03-23)
- [x] **Calendar/Email = text reference:** Local text reference only, no API integration. (confirmed 2026-03-23)
- [x] **SOP‚ÜíTask = one-way:** Task references which SOP it follows. SOPs don't generate tasks. (confirmed 2026-03-23)
- [x] **Sync = automated + Claude-assisted:** Automated where possible, Claude fills gaps. (confirmed 2026-03-23)
- [x] **Conflict resolution = timestamp-based:** No single master. Last-write-wins using timestamps. (confirmed 2026-03-23)
  - Each task stores: `Created:` (when first written), `Updated:` (last local edit), `ext:` (external system ID), `synced:` (last sync time)
  - `Updated > synced` ‚Üí push local to external system
  - `External updatedAt > synced` ‚Üí pull external to local
  - Both changed since last sync ‚Üí **conflict** ‚Üí Claude flags for manual decision
- [x] **Timestamps format:** `Created:` and `Updated:` use `YYYY-MM-DD`. Only `synced:` uses full `YYYY-MM-DDTHH:MM` for conflict precision. (confirmed 2026-03-23)
- [x] **Timestamp line position:** Always line 4 (after GTD labels, core metadata, before ontology edges). Always present on every task. (confirmed 2026-03-23)

- [x] **Goal cascade = shallow:** Just a tag on the task (`Goal: Q1-ship-[diagnostic-service]`). No KR/Initiative hierarchy. (confirmed 2026-03-23)
- [x] **External system = per-project default + per-task override:** Each project has a default sync target ([Task Manager], Asana, Trello, etc.). Tasks inherit the project default. Individual tasks can override with a different system reference. (confirmed 2026-03-23)

### Per-task sync fields:
```markdown

  - sync: asana          ‚Üê which system (inherits project default if omitted)
  - ext: 000000000000  ‚Üê external system ID for this task
  - synced: 2026-03-23T14:30
  - updated: 2026-03-23T16:45
```

### Supported systems:
- `[task-manager]` ‚Äî [Task Manager] (API via MCP)
- `asana` ‚Äî Asana (API via MCP)
- `trello` ‚Äî Trello (manual or API)
- `bitrix` ‚Äî Bitrix24 (manual or API)

### Rules:
- Project-level `sync:` field in TASKS.md frontmatter = default for all tasks
- Task-level `sync:` overrides project default (rare ‚Äî e.g., one task tracked in client's Bitrix while rest is in [Task Manager])
- `ext:` = the task ID in the external system (Asana task ID, [Task Manager] task ID, Trello card ID, etc.)
- Conflict resolution uses same timestamp logic regardless of external system

## 12. Task Types (9 types, confirmed 2026-03-23)

| Type | Label | Has action? | Has date? | Has owner? |
|---|---|---|---|---|
| Action | `@next` / `@computer` / `@phone` / `@email` | Yes | Usually | Yes |
| Reminder | `@reminder` | No | Yes (trigger date) | Optional |
| Waiting | `@waiting` | No (someone else) | Optional (follow-up) | Yes (who) |
| Monitor | `@monitor` | Check, not do | Yes (check date/frequency) | Optional |
| Decision | `@decision` | Think, not do | Usually (deadline) | Decision maker |
| Someday | `@someday` | Maybe later | No | No |
| Reference | `@reference` | No | No | No |
| Milestone | `@milestone` | No | Yes (fixed date) | No |

### Monitor subtypes (all use `@monitor`, distinguished by context):
- **Post-fix verification** ‚Äî did the fix work? is it stable?
- **Platform health** ‚Äî fragile thing, check after changes
- **Spend/performance drift** ‚Äî budget or metric going off track
- **External dependency** ‚Äî something outside your control that could break you
- **Compliance** ‚Äî legal/consent things that must stay clean

### Monitor fields:
```markdown
- [ ] Task description
  - @monitor
  - Check: what to look at and where
  - Baseline: starting value/state (date)
  - Target: what "good" looks like
  - Risk: what could go wrong
  - Frequency: when to check (weekly / after deploys / 2x/week / etc.)
```

## 13. Effort & Impact Scales (confirmed 2026-03-23)

### Effort = time
`15m` | `1h` | `2h` | `4h` | `1d` | `2d` | `3d` | `1w` | `2w+`

### Impact = business outcome (5 levels)
| Impact | Meaning |
|---|---|
| `noise` | Doesn't move anything meaningful |
| `hygiene` | Prevents problems, no upside ‚Äî avoids downside |
| `lever` | Multiplies something already working |

### Usage in task:
```markdown

  - P1 | Owner: [Owner] | Due: 2026-03-28 | Effort: 15m | Impact: lever
```

## 14. File Structure (confirmed 2026-03-23)

Each project gets TWO files:
- `TASKS.md` ‚Äî active tasks only (P0-P3 + Reference). Expected: 100-300 per project.
- `TASKS_DONE.md` ‚Äî completed + wont-do tasks. Append-only archive.

When a task is completed:
1. Move it from `TASKS.md` to top of `TASKS_DONE.md` (newest first)
2. Add completion date: `- [x] Task description (2026-03-23)`
3. Keep all metadata (ontology edges, effort, impact) ‚Äî it's historical record

`TASKS_DONE.md` frontmatter:
```yaml
---
project: "project-name"
type: archive
updated: 2026-03-23T00:00
---
```

Rules:
- Never delete from TASKS_DONE.md
- TASKS.md stays lean ‚Äî only what needs attention
- Reference section stays in TASKS.md (it's active context, not done)
- Sync metadata (`ext:`, `synced:`, `updated:`) preserved in archive for audit trail
